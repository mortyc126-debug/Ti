# BLUEPRINT — БондАналитик

Большая техзаписка для всех веток разработки. Читать **первой**, прежде чем
браться за конкретный track. Описывает видение, текущее состояние, слои
системы, контракты данных и параллельные ветки работ.

Все ветки разработки начинаются от `claude/wizardly-feynman-64TbH` и
сливаются обратно в неё. Файлы не пересекаются (см. таблицу треков ниже),
поэтому merge clean.

---

## 1. Видение

Один пользователь — частный инвестор в российские корпоративные облигации
(преимущественно ВДО) и фьючерсы — хочет видеть **полную картину
эмитента** и **поведение его бумаг в моменте**, чтобы:

1. **Оценивать платёжеспособность** — не только текущую (по показателям),
   но и в схожих экономических условиях из прошлого.
2. **Распознавать стресс заранее** — по динамике цены, объёмов,
   волатильности, спредов и событий.
3. **Прогнозировать кассовые разрывы** — у каких эмитентов в ближайшие
   12 месяцев платежей больше, чем способно покрыть EBITDA + cash.
4. **Находить похожих** — для оценки риска по когорте: «как другие в той
   же отрасли, с такой же долговой нагрузкой и того же размера прошли
   2014/2020/2022». Это контекст, без которого голая метрика бессмысленна.
5. **Ловить моменты входа/выхода** для фьючерсов — через стакан, объёмы,
   баланс заявок. Для акций и облигаций — те же индикаторы, но с
   терпимостью к ~15-минутной задержке MOEX.

**Ключевая ценность системы — связки**, которые в платных терминалах
требуют отдельных подписок: фундаментал × поведение цены × события ×
макро × отрасль × холдинговые связи.

## 2. Текущее состояние (на коммит `26f3a78`, ветка `claude/wizardly-feynman-64TbH`)

### Уже работает

**Backend (Cloudflare Worker `bondan-backend`, D1 `coldline`):**
- `MOEX ISS` коллекторы: акции (TQBR), фьючерсы (FORTS),
  облигации (TQCB/TQOB/TQOD/TQOY/TQED). Ежедневный snapshot, cron.
- Справочник эмитентов через MOEX bulk: ~1136 ИНН, классификация
  `kind = corporate | bank | subfederal | municipal | federal`.
- РСБУ-отчётность: каскад **buxbalans → ГИР БО → Grok**. ~1300+ строк
  на 119+ эмитентов. Ожидаемый год + cooldown'ы +30/+7/+14/+90.
- Аффилиации: DaData free → имя/статус/сектор/директор. zachBiz fallback
  → ИНН учредителей-юрлиц. Цепочка холдингов раскрывается за 2-3 прохода.
- AI-каскад: Cerebras (архивы) + Grok (текущие, без Live Search) +
  ai_cache (TTL 30 дней) + ai_calls_log.
- Endpoint'ы: `/status`, `/catalog`, `/bond/latest`, `/issuer/{inn}`,
  `/issuer/{inn}/reports`, `/issuer/{inn}/affiliations`, `/diag/dadata`,
  `/collect/{stock|futures|bonds|issuers|reports|affiliations}`,
  `/ai/extract` (engine=cerebras|grok).

**Frontend (`web/`, React + Vite, на CF Pages):**
- Главная (KPI, события — мок), Облигации (24 бумаги — мок), Избранное,
  Live, Портфель. **Реальные данные пока подключены только к
  `/status`/`/catalog`** — таблица облигаций показывает мок.

**Клиентский standalone (`index.html`):** старое монолитное SPA, работает
с localStorage. Содержит ручные импорты, парсеры PDF/XLSX, страницу
сравнения когорт. Используется как образец логики (например, маппинг
ОКВЭД→отрасль, регексы для аудит-ит).

### Критические ограничения

- **Cloudflare Workers free tier**: 50 subrequest на вызов, 30 сек
  wall-clock, 10ms CPU. Все коллекторы работают с этими лимитами в уме
  (пагинация + time budget). Платный план ($5/мес) снимет ~95% боли,
  но решение пока «нет».
- **e-disclosure.ru, audit-it.ru, list-org.com** — недоступны
  (servicepipe / cerb / CF block). Нужен либо коммерческий
  web-unblock (~$30/мес), либо Browserless headless.
- **DaData free** не отдаёт founders[]/managers[]/finance/predecessors —
  только в CleanerAPI (~3000₽/мес). Поэтому zachBiz парсится дополнительно.
- **ГИР БО** часто блокирует CF Worker IP по гео — есть auto-disable.

### Контракт данных (общий для всех веток)

Все цифры — в **млрд ₽** во внутренних таблицах (`issuer_reports.rev`,
`debt`, и т.д.). Внешние источники могут давать тыс/млн — преобразовать
при INSERT.

Все ИНН — `TEXT` (10 или 12 цифр строкой, не number — есть префиксные нули).

Все даты — ISO `YYYY-MM-DD`. Таймстампы — ISO `YYYY-MM-DDTHH:MM:SSZ` (UTC).

Цена бумаг — `REAL`, в % от номинала для облигаций; в рублях за акцию;
в пунктах для фьючерсов (с stepPrice для конвертации в рубли).

## 4. Параллельные треки

Каждая ветка работает в **отдельной зоне worker.js** между маркерами
`// ═══ TRACK X: <name> ═══` и `// ═══ END TRACK X ═══`. Новые
collector-функции, endpoint'ы и расчёты добавляются внутри этих зон.
**Никто не редактирует чужую зону.** Schema migrations — отдельные
файлы `backend/migrations/<track>_*.sql`, идемпотентные.

| # | Track | Файлы | Зависит от | Сложн. | Сессий |
|---|---|---|---|---|---|
| **A** | Order book / FORTS intraday | `backend/orderbook.js`, schema | — | средн | 2-3 |
| **B** | Макро-индикаторы | `backend/macro.js`, schema | — | низк | 1-2 |
| **C** | Корпоративные события | `backend/events.js`, schema | — | высок | 3-4 |
| **D** | Рейтинговые действия | `backend/ratings.js`, schema | — | средн | 2 |
| **E** | Стресс-индекс + когорты | `backend/stress.js`, schema | B (мягко) | высок | 3 |
| **F** | Default predictor (LLM) | `backend/risk.js` | C, D, E | средн | 2 |
| **G** | Frontend на реальные данные | `web/src/**/*.{jsx,js}` | (ничего из бэка) | высок | 3-4 |
| **H** | Банковская отчётность ЦБ | `backend/cbr-bank.js`, schema | — | средн | 2 |
| **I** | Telegram-бот алертов | `backend/telegram.js` (отдельный воркер) | C, E (мягко) | средн | 2 |

**Всего ~9 треков, ~22 сессии.** Можно запустить минимум **5 параллельных
веток** (A, B, D, G, H — без зависимостей), три после первого слива
(C, E, F + I).

Полный brief каждого трека — в `docs/tracks/{A..I}_*.md`.

## 5. Конвенции (общий протокол слияния)

См. `docs/CONVENTIONS.md` — но кратко:

1. **Одна ветка = одна зона.** Никаких правок чужих секций.
2. **Schema только новыми таблицами.** Никаких ALTER existing — кроме
   через worker.js auto-migrate в catch'е (пример — `kind` в issuers).
3. **Endpoint'ы только новыми.** Не менять контракт существующих
   (только добавлять необязательные query-параметры).
4. **Версия в /status.** Каждый коммит обновляет `version` строкой
   `0.X.Y-track-name` чтобы проверить, что задеплоилось.
5. **Идемпотентность.** Все коллекторы должны быть безопасны для
   повторного запуска (UPSERT, IF NOT EXISTS, COALESCE).
6. **Time budget.** Все коллекторы поддерживают `?max_ms=N` и
   проверяют `Date.now() - t0 > maxMs - 2000` перед каждой операцией —
   чтобы укладываться в 30 сек free tier.
7. **Subrequest budget.** Не более 50 fetch() на вызов на free tier,
   1000 на paid. Документировать ожидаемое количество в комментарии.
8. **Auto-migrate.** Все ALTER TABLE / CREATE TABLE в коллекторе
   обёрнуты в try/catch — на уже существующих БД работают.
9. **Тесты в комментариях.** В каждом нетривиальном парсере оставить
   1-2 примера ожидаемого input → output.
10. **Версия Worker.js bumpAется только в самом конце коммита**, чтобы
    в /status сразу было видно, какая ветка задеплоена.

## 6. Приоритеты и стратегия слияния

### Phase 1 (готово к параллельному запуску):

- **A (orderbook)** — нужен пользователю для фьючерсов в реальном времени.
- **B (macro)** — фундамент для стресс-индекса, простой и быстрый.
- **D (ratings)** — лёгкая ветка, лучшие первые «события».
- **G (frontend)** — пользователь сейчас видит мок-данные, это
  блокирует UX.
- **H (cbr-bank)** — нужен для 107 банков, у которых сейчас 0
  отчётности.

Эти 5 веток **не зависят друг от друга** и от уже существующего кода
сверх контракта схемы. Можно стартовать одновременно.

### Phase 2 (после слияния Phase 1):

- **C (events)** — нужна оптимизация (CF / Cerebras для триажа +
  возможно платный web-unblock). После B готова — макро-фон даёт
  контекст событиям.
- **E (stress)** — нужны данные A, B, D и существующие котировки.
- **I (telegram)** — нужны C+E чтобы было на что алертить.

### Phase 3:

- **F (risk)** — финальная LLM-обвязка, нужны все остальные.
- **Интеграционная неделя** — связать всё в единые витрины.

## 8. Что НЕ делаем (принципиально)

- **Не строим биржу-копию.** Не предсказываем точные цены.
- **Не торгуем авто.** Только подсказки.
- **Не покупаем data — пока.** Все источники бесплатные или у пользователя
  уже есть. Платные только если ROI очевиден (например, $30/мес за
  Browserless вернёт 200 эмитентов из e-disclosure → x2 покрытие).
- **Не пишем длинных AI-промптов.** Cerebras/Grok вызываем точечно с
  чётким JSON-выводом, не делаем «помоги с инвестициями» chat.
- **Не заводим UI-фичи без бэкенд-данных.** Сначала truth, потом view.

## 9. Как начать новой ветке

1. `git fetch origin && git checkout claude/wizardly-feynman-64TbH`
2. Создать свою ветку `claude/track-X-<slug>` от текущего коммита.
3. Прочитать `docs/tracks/<X>_*.md` — там точная цель, схема, endpoints.
4. Прочитать `docs/CONVENTIONS.md` — формат коммитов, тестирование.
5. Сделать всю работу в **отдельной зоне** worker.js (с маркерами).
6. **Не править** существующие функции — только дополнять.
7. Bump `version` в `/status` на `0.X.Y-track-N`.
8. Push в свою ветку. Pull request → main branch (`claude/wizardly-feynman-64TbH`).

Точка входа в общую картину — этот файл. Если что-то противоречит
этому файлу, **противоречие в файле — обновляйте**, не code.
