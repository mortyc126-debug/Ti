# Системный прогон — цельная оценка живого стека

Кнопка **🧭 СИСТЕМНЫЙ ПРОГОН** в дашборде (панель «Бэктест по тикерам»).

## Зачем

Обычный бэктест дашборда всегда гоняет `OICompositeStrategy` — он НЕ отражает,
как система торгует после разделения стратегий по ликвидности
(`[FUTURES_STRATEGY_MAP]`: IRAO/PLZL→NW, остальной фронт→accel по
`STRATEGY_OVERRIDE`). Системный прогон оценивает **каждый тикер через ту
стратегию, которой он торгует вживую**, и сводит всё в одну таблицу —
composite + accel + NW рядом, сопоставимо.

## Как считает (честно)

Единый бар-за-баром симулятор `system_backtest.simulate_analyze_strategy`,
общий для всех стратегий (все — `IStrategy` с `analyze_candles`, т.е. гоняется
ровно то, что работает вживую):

- **held-out**: первые `split_frac` (по умолчанию 60%) свечей — прогрев
  (train: accel наполняет буфер, NW строит память, composite греет
  перцентили), сигналы считаются только на оставшихся 40% (OOS).
- **вход** по `Signal` LONG/SHORT рынком на закрытии бара; **выход** интрабар
  по тейк/стопу из самого сигнала ИЛИ по `CLOSE` (срочный выход NW через 12
  баров). Без перекрытия — одна позиция на инструмент.
- **метрика**: экспектанси в ATR входа за вычетом `cost_atr` (по умолчанию
  0.12), winrate, N сделок. Свод по стратегиям — суммарный N, взвешенная по N
  экспектанси.

Это отражает живое поведение, а НЕ родной `backtest_barriers` композита
(другая модель выхода) — цель именно «как система торгует», в одной шкале.

## Методы (композит с пресетом) — сравнение/комбинирование

Рядом с кнопкой — селектор **«+ методы»**. Если выбрать пресет (из
`method_presets.json`) или «текущий toggle», прогон добавляет по каждому тикеру
ВТОРУЮ строку: `OICompositeStrategy` с этим набором `disabled`/`inverted`
(применяется через `set_disabled_methods`/`set_inverted_methods`), прогнанную
тем же симулятором и в той же шкале (exp ATR). Так accel/NW и методы видно в
одной таблице — можно сравнивать и комбинировать, менять пресет и перегонять.

- Живая строка тикера — его боевая стратегия (accel/NW). Композит-строка —
  референс с отступом. Свод `by_strategy` считает accel / NW / `composite·пресет N`
  раздельными вёдрами.
- Если живая стратегия тикера сама composite — пресет применяется прямо к ней
  (второй строки нет).
- Это оценка/сравнение, НЕ блендинг в одну позицию: у стратегий разные выходы,
  смешивать их в одну сделку валидно только после отдельного гонтлета.

## Запуск

- В дашборде: кнопка + селектор «+ методы» (по умолчанию 90 дней свечей).
- API: `GET /api/system_backtest?days=90` (опц. `&tickers=IRAO,PLZL,GAZP`,
  `&preset=<имя из method_presets.json | current>`).
- Отдаёт `{rows:[{ticker, strategy, kind:'live'|'composite', n, win, exp_atr}],
  by_strategy:{...}, preset_label}`.

## Оговорки

- **Нужны данные**: считается там, где кэш свечей (сервер), не в пустом окружении.
- Метрика exp_atr сопоставима между стратегиями по шкале, но выходы у них
  разные по природе (accel: тейк/стоп 1.0/0.3; NW: срочный 12 баров;
  composite: адаптивный) — сравнивать разумно, но не в лоб «кто больше».
- Для composite учитывается текущий пресет методов (`method_toggle_state.json`).
- Это оценка живой раскладки, а не перебор параметров; для тюнинга NW —
  по-прежнему `nw_memory.py --batch` по кэшу (per-ticker walk-forward).

## Код

- `system_backtest.py` — `live_strategy_name` (роутинг как в трейдере: карта >
  override > дефолт) + `simulate_analyze_strategy` (симулятор). Чистый модуль,
  тестируется на синтетике без сети.
- `dashboard.py` — `run_system_backtest` (оркестрация: свечи, диспетчер,
  свод) + эндпоинт `/api/system_backtest` + кнопка/таблица.
