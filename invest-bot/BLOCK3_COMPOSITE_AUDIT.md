# Блок 3. Композит методов — логика, конфликты, синергии

Аудит взвешенной суммы 50+ методов в `OICompositeStrategy`: где методы
дублируют друг друга, где противоречат по конструкции, и где один и тот же
сигнал считается дважды и раздувает размер позиции.

Все ссылки — на `trade_system/strategies/oi_composite_strategy.py`, если не
указан другой файл.

## 0. Как устроена сумма (карта пайплайна)

`__compute_composite` (≈8195–8375):

```
scores (ALL_METHOD_NAMES)
  → перцентильная нормализация (calibrator)
  → alt-трансформации (дивергенция/истощение)          # _apply_alt_transforms
  → × regime_mods           (Layer 2, REGIME_WEIGHT_MODS / динамика)
  → × redundancy_mult       (Layer 4, redundancy_dampen по |corr|)
  → × IC-инверсия / IC-байес / hedge / phase / micro-boost
weighted = Σ score·w   по BASE_METHOD_NAMES  (M1/M2/M3 исключены)
linear_raw = weighted / Σ|w|
  → + div_score·0.25                                    # ещё раз base-методы
  → blended = 0.6·playbook + 0.4·linear   если |playbook|>0.08
  → × (0.6+0.4·vhf) × rqa × wavelet × regime_conf × lag
  → composite = 0.7·L3 + 0.3·L2           (_MTF5_BLEND_W)
confidence = 0.5 + 0.5·|composite|   → бакет риска → размер
```

Ключевое: `confidence = 0.5 + 0.5·|composite|` (9225), а размер — **ступенчатая**
функция confidence (`risk_config.py`):

| \|composite\| | confidence | риск от депо |
|---|---|---|
| < 0.10 | < 0.55 | вход запрещён |
| 0.10–0.40 | 0.55–0.70 | **0.5 %** (RISK_MIN) |
| 0.40–0.70 | 0.70–0.85 | **1.0 %** (RISK_MID) |
| ≥ 0.70 | 0.85+ | **1.5 %** (RISK_MAX) |

Границы на `|composite| = 0.10 / 0.40 / 0.70`. **Любой шаг composite,
пересекающий 0.40 или 0.70, удваивает/полуторакратит размер.** Это усилитель
для всех эффектов двойного счёта ниже: важна не «косметическая» добавка к
composite, а то, пересекает ли она границу бакета.

## 1. Корреляционный аудит (redundancy) — инструмент готов, нужен прогон

`redundancy_analysis.py` в этом окружении **не запускается**: нет токена
Tinkoff, нет кэша свечей (`data/`), не установлены numpy/scipy. Плюс он и не
печатает *попарные* корреляции — только средний `|corr|` метода с кластером,
чего для «какие пары >0.7» недостаточно.

**Сделано:** добавлен `redundancy_from_history.py` — офлайн-версия, которая
читает `data/history.json` напрямую (без Tinkoff, numpy опционален, есть
pure-Python Pearson) и выводит **попарные** corr внутри кластера >0.7 +
пометку, у кого из пары слабее edge (кандидат на удаление):

```
python redundancy_from_history.py data/history.json --source both --threshold 0.7
python redundancy_from_history.py --source trades --by-regime   # разрез по режимам
```

Проверен на синтетике — ловит пары и агрегирует по тикерам. Источник рядов:
`daily` (дневные снэпшоты — то же, что питает живой `redundancy_dampen`),
`trades` (скоры на входе — сюда попадают «виртуальные сделки»), `both`.

**Про твои данные:** большой до-фиксовый набор **годится** для пула
корреляций — формулы методов при фиксе режимов не менялись, фикс поменял
только *ярлык* режима. Испорчен там только разрез `--by-regime` (старые
ярлыки). Поэтому: гони большой набор без `--by-regime` для списка «пар >0.7»,
свежий маленький — для sanity-check.

**Структурные кандидаты на дубль (проверить прогоном в первую очередь)** —
из формул, до данных:

| Кластер | Пара | Почему близки |
|---|---|---|
| Тренд | `PRICE_TREND` ~ `TREND_QUALITY` ~ `SSA_SIGNAL` | все три — знак/сила направленного движения по окну |
| Адаптивные МА | `ADAPTIVE_MA` ~ `ZLEMA_SIGNAL` ~ `T3_SIGNAL` | отклонение цены от сглаживающей МА с памятью |
| Импульс/Осц. | `RMI` ~ `FISHER_RSI` | RSI-семейство, оба перевозбуждаются синхронно |
| Объём | `KLINGER` ~ `TWIGGS` ~ `VZO` | объём×направление, один физический поток |
| Позиционирование | `INST_OI` ~ `RETAIL_CONTRA` | зеркало юр/физ из одного FutOI (см. п.4) |
| Микроструктура | `BS_PRESSURE_TS` ~ `AGGRESSOR_FLOW` | давление/агрессия из одного tradestats-среза |

`redundancy_dampen` (Layer 4) их не *удаляет*, только множит вес на
`max(0.3, 1−|corr|)` — при corr=0.9 остаётся 0.3 у каждого, т.е. пара всё
равно даёт ~0.6 «одного» голоса вместо 1.0. Для явных дублей (corr>0.85 в
прогоне) дешевле убрать один из METHODS, чем полагаться на демпфер.

## 2. Трендследящие vs контр-трендовые: где гасят, где шумят

**Главная находка: режимный флип знака ОТКЛЮЧЁН.** `_REGIME_METHOD_SIGN = {}`
пусто (306), ветка на 8253 (`if regime in _REGIME_METHOD_SIGN`) **никогда не
срабатывает** (комментарий: константа глобальная, ломала тикеры). Значит
единственное, что разводит трендовые и контр-трендовые — это *модуль* веса в
`REGIME_WEIGHT_MODS` (regime.py 41–122). Знак у методов **всегда сырой**:
ZSCORE в сильном аптренде читает «перекуплено → SHORT» и **вычитается** из
composite своим весом — не «выключается», а активно тянет против тренда.

Веса по режимам (трендовые / контр-трендовые):

| режим | PRICE_TREND | ADAPTIVE_MA | DONCHIAN | ZSCORE | FISHER_RSI | SINEWAVE |
|---|---|---|---|---|---|---|
| trending_up/down | **1.6** | 1.4 | 1.4 | 0.8 | 0.8 | 0.7 |
| ranging | 0.5 | 0.6 | 0.6 | **1.2** | **1.2** | **1.3** |
| high_vol | 0.5 | 0.6 | 0.7 | 0.9 | 0.9 | 0.8 |
| low_vol | 0.8 | 0.8 | 0.8 | 1.1 | 1.1 | 1.2 |
| **stress** | **0.3** | 0.6 | **1.0*** | **1.0*** | **1.0*** | **1.0*** |

`*` — метод **не указан** в dict режима stress → откат на 1.0 (regime.py 113–121).

**Где реально гасят друг друга:** только за счёт разницы модулей. В тренде
трендовые 1.6 против контр 0.8 — соотношение 2:1, контр-тренд остаётся
0.8-весовым встречным ветром, не гасится в ноль. В боковике наоборот. Полного
взаимопогашения нигде нет — обе группы всегда в сумме.

**Где создают шум (конкретно):**

1. **stress — осцилляторы на полном весе.** В панике трендовые прибиты до 0.3,
   но ZSCORE/FISHER_RSI/SINEWAVE **не перечислены** в dict stress → работают на
   1.0. Ловля отскока mean-reversion-методом на полном весе в панике — это
   ловля падающего ножа. Скорее всего забыли дописать, а не задумано (комментарий
   гласит «паника ломает любые паттерны», но приглушили только тренд/структуру).

2. **DONCHIAN — непоследователен.** Трендследящий пробой, но:
   (а) в stress не приглушён (→1.0), хотя PRICE_TREND=0.3;
   (б) задокументирован как эмпирически инвертируемый (`_EMPIRICAL_INVERTED_METHODS`,
   250) — т.е. по статистике его «согласный» голос = антисигнал (WR 35 %);
   (в) при этом в трендовых режимах ему дают 1.4. Итог: сильный вес методу,
   чей знак под вопросом.

3. **`redundancy_dampen` бьёт по |corr| — гасит и противников, и дубли
   одинаково.** Трендовый up-vote и mean-reversion down-vote *анти*коррелированы
   (corr→−1), но демпфер берёт `abs` (cluster_models 366/397) → оба получают
   `max(0.3, 1−0.7)=0.3`. Т.е. система уже глушит противостоящую пару до 30 % —
   но это глушит и *информацию* о расхождении, которую отдельно пытаются достать
   `_divergence_score` и плейбуки. Двойная работа с противоположным знаком
   намерения.

**Рекомендации п.2:**
- Дописать stress-режим для ZSCORE/FISHER_RSI/SINEWAVE (например 0.5–0.6, как
  соседние), иначе в самом опасном режиме контр-тренд самый громкий.
- Решить судьбу DONCHIAN: либо снять трендовый буст 1.4, либо явно инвертировать
  (сейчас `_EMPIRICAL_INVERTED_METHODS` — мёртвая константа, см. доп. находки).
- Рассмотреть замену `abs(corr)` в `redundancy_dampen` на асимметрию: дубли
  (corr>0) демпфировать, анти-корреляцию (corr<0) — не глушить, а отдавать в
  divergence-канал.

## 3. Двойной счёт: плейбуки поверх тех же base-методов

`_compute_playbooks` (3317–3502) — 7 конъюнкций. **Все они собраны из тех же
base-методов, что уже просуммированы в `linear_raw`.** M1/M2/M3 защищены
комментарием «во избежание двойного счёта» (6000–6007) и исключены из суммы —
но у плейбуков такой защиты нет: при активации плейбук берёт **60 %** итога
(`blended = 0.6·playbook + 0.4·linear`, 8340).

Что из чего собран (все компоненты — уже в линейной сумме):

| Плейбук | base-методы внутри |
|---|---|
| ABSORPTION | HAWKES + VSA + LEVEL_CONTEXT (+AGGRESSOR) |
| FAKEOUT | SPRING + WICK + LEVEL + OI_SQUEEZE |
| REGIME_SHIFT | CHANGE_POINT + SINEWAVE + MKT_STRUCTURE + FRACTAL |
| CONSOLIDATION_BREAK | TRIANGLE + OI_SQUEEZE + INST_OI + RETAIL |
| TREND_PULLBACK | TREND_QUALITY + FRACTAL + VWAP + VOL_MOMENTUM + CANDLE |
| EXHAUSTION_DIV | PRICE_TREND + VOL_MOMENTUM + HAWKES + RMI/FISHER + MULTI |
| **OI_CONSENSUS** | OI_SQUEEZE + INST_OI + RETAIL + DELTA_QUADRANT + OI_ABSORPTION |

**Вес двойного счёта в цифрах.** Плейбук не *добавляется* к linear, а
*замещает* 60 % итога. Пример для OI_CONSENSUS (пятёрка OI-методов, доля в
linear ≈ 5/30 ≈ 0.17 при сопоставимых весах):

```
OI_mean=0.30 → вклад OI в composite: без плейбука ~0.051, с плейбуком ~0.254  (×5.0)
OI_mean=0.50 → ~0.085 → ~0.394  (×4.6)
OI_mean=0.70 → ~0.119 → ~0.408  (×3.4)
```

То есть активный плейбук усиливает вклад своих base-методов в composite в
**3.4–5 раз** относительно их честной линейной доли. Для OI_mean=0.5 это тянет
composite к 0.39 — вплотную к границе бакета 0.40 (0.5 %→1.0 % риска). Причём
`redundancy_dampen` (Layer 4) правит **только** линейную часть, плейбук он **не
трогает** — вся защита от избыточности обходится стороной по 60 %-каналу.

Важно: это не баг конъюнкций (нелинейное «крупный агрессор поглощает на уровне»
действительно > суммы частей). Проблема в **фиксированных 0.6/0.4**: 60 % отдаётся
плейбуку независимо от того, сколько *независимых* методов реально сложились.
OI_CONSENSUS честно снижает свой *скор* до cap 0.6 (комментарий 3472: «не пять
свидетелей, а пять взглядов на одну таблицу»), но **не** свою 60 %-долю блендинга.

**Рекомендации п.3:**
- Масштабировать долю блендинга числом *независимых кластеров* в плейбуке:
  OI_CONSENSUS (один кластер «Позиционирование») → доля 0.6·(1/N_кластеров), а
  ABSORPTION (HAWKES-воло + VSA-объём + LEVEL-структура, 3 кластера) → полная.
- Либо вычитать вклад base-методов плейбука из linear перед блендингом (как
  сделано для M1/M2/M3), чтобы 60 % не накладывались на их же линейную долю.
- `calibrate_playbook_activation_levels` в joint_calibration.py калибрует пороги
  активации, но не долю 0.6 — её тоже стоит завести в калибровку.

## 4. OI_CONSENSUS × signal_gate: одна FutOI-картина, тройной учёт

Оба читают один и тот же `data/oi_daily.json` (FutOI физ/юр). Но роли разные:

- **OI_CONSENSUS** (плейбук) — *добавляет* в composite (см. п.3), значит через
  `confidence` *влияет на размер*.
- **signal_gate.evaluate** (signal_gate.py 680–705; вызов trader.py 1318) —
  *жёсткое вето*, только `(ok, reason)`, только шорты, только «не входить».
  **Разжать размер он не может — только заблокировать.**

Поэтому формулировка «signal_gate инфлирует размер» неточна: **инфляции размера
от гейта нет**. Реальная инфляция — от OI_CONSENSUS (п.3). Но есть более тонкая
проблема, которую стоит зафиксировать:

**Иллюзия независимого подтверждения.** Одна FutOI-картина учитывается **трижды**:
(1) 5 OI-методов в линейной сумме; (2) OI_CONSENSUS-плейбук ×3.4–5 поверх них;
(3) signal_gate как «независимый» фильтр. Когда FutOI-сетап бычий, все три
срабатывают согласованно **не потому что три источника согласны, а потому что
источник один**. Внутри самой пятёрки INST_OI и RETAIL_CONTRA почти зеркальны
(комментарий 3474) — фактически «независимых» OI-взглядов там 2–3, не 5.

Итог: размер раздувается по каналу (1)+(2) — от одной таблицы, — а (3) создаёт
ложное ощущение, что сетап ещё и «прошёл независимый гейт». По границам бакета
(0.40/0.70) этого достаточно, чтобы одна FutOI-картина в одиночку подняла риск
с 0.5 % до 1.0 %.

**Рекомендации п.4:**
- Держать «OI-бюджет» на весь FutOI-кластер: суммарный вклад OI-методов +
  OI_CONSENSUS в composite ограничить одним потолком (напр. как один усиленный
  метод, не как 5+плейбук).
- Явно НЕ считать прохождение signal_gate подтверждением при активном
  OI_CONSENSUS (это тот же сигнал) — в attribution/логах пометить как
  зависимый, чтобы калибровка не переоценивала «сходимость».
- Схлопнуть INST_OI/RETAIL_CONTRA в один метод или дать паре общий вес (они
  зеркальны — прогон п.1 покажет corr, ожидаемо >0.85).

## 5. MTF-блендинг: `_MTF5_BLEND_W = 0.30` — круглое число, не калибровка

`composite = 0.7·L3 + 0.3·L2` (8358). Проверено: константа `_MTF5_BLEND_W`
встречается **только** в определении (5866) и в этой строке. Её **не трогают**
ни joint_calibration.py, ни method_calibrator.py — не калибруется ни статически,
ни онлайн. Комментарий не даёт WR-обоснования (в отличие от весов
REGIME_WEIGHT_MODS, где стоит «за=55 % WR»). Это **круглое число**.

**Чувствительность** (линейный бленд, `d(composite)/dw = L2 − L3`):

```
   L3    L2 | w=0.20 w=0.30 w=0.40   Δ(0.2→0.4)
 +0.50 +0.50 | +0.500 +0.500 +0.500   +0.000    ТФ согласны → w не важен
 +0.50 +0.20 | +0.440 +0.410 +0.380   −0.060
 +0.50 −0.20 | +0.360 +0.290 +0.220   −0.140
 +0.50 −0.50 | +0.300 +0.200 +0.100   −0.200    ТФ спорят → w решает
 +0.35 −0.10 | +0.260 +0.215 +0.170   −0.090
```

Вывод: **вес L2 важен ровно тогда, когда ТФ расходятся** — а это и есть момент,
когда решение самое хрупкое. При L3=+0.5/L2=−0.5 сдвиг w с 0.2 на 0.4 двигает
composite на −0.20 — это заведомо пересечение границы бакета (вход/нет-входа,
0.5 %/1.0 %). То есть цена «круглости» 0.30 — прямое влияние на размер и на сам
факт входа в спорных ситуациях. Плюс поверх ещё висит `_l2_momentum_mult` (×0.65)
и fallback-ветка ZLEMA — т.е. L2 уже влияет несколькими путями.

**Рекомендации п.5:**
- Завести `_MTF5_BLEND_W` в joint_calibration (grid 0.20/0.25/0.30/0.35/0.40 по
  IR/quality на истории) — это дешёвый одномерный подбор.
- Либо сделать w адаптивным: при согласии ТФ w не важен, при расхождении вес L2
  логично привязать к его надёжности (числу 5м-баров `_MTF5_MIN_5M_BARS` уже
  есть, но в бленд не входит) — сейчас 12 баров L2 и 100 баров L2 весят одинаково 0.30.

## 6. METHOD_TF_CONFIG: половина весов 5м без обоснования

`METHOD_TF_CONFIG` (5872–5913) — `weight_5m` на 5-минутках. Часть весов подписана
причиной, часть — просто круглые 1.05–1.10:

**С обоснованием** (оставить как есть до прогона):
`TREND_QUALITY 1.20` (HH/HL меньше шума), `DONCHIAN 1.10` (структура боковика),
`CUMUL_DELTA 1.20` (агрессия надёжнее), `AMT_POC 1.10` (POC-сессия),
`VSA_ABSORPTION 1.15`, `CASCADE 1.25`, `IMPULSE_PULLBACK 1.15`, `PRICE_ACCEL 1.10`,
`FRACTAL 1.25` (Hurst нужно много баров), `VWAP_SIGNAL 0.85` (1м точнее),
`SINEWAVE 0.80`/`CYBER 0.80`/`EBSW 0.85` (циклы нестабильны), `ZSCORE 0.95`,
`RMI/FISHER 1.05`, `CANDLE/WICK 1.15`, `HAWKES 1.30` (микроструктура ведущая).

**Без обоснования — кандидаты на перекалибровку** (просто 1.05–1.10 в общей
группе):
`PRICE_TREND 1.10`, `ADAPTIVE_MA 1.10`, `ZLEMA 1.05`, `T3 1.05`,
`VOL_MOMENTUM 1.10`, `KLINGER 1.10`, `VZO 1.05`, `TWIGGS 1.05`,
`MA_ENVELOPE 1.10`, `NADARAYA_WATSON 1.10`, `FRACTIONAL_DIFF 1.10`,
`VSA 1.05`, `ENTROPY 1.10`.

Все 13 «безымянных» сидят в узком коридоре 1.05–1.10 — это дефолтный «чуть выше
нейтрали», а не измеренное преимущество на 5м. Их надо гнать через
`lag_analysis.py` (стабильность лага на 5м) + `redundancy_from_history.py` (не
дублируют ли уже учтённое). Особый случай — `NADARAYA_WATSON`/`FRACTIONAL_DIFF`:
в `REGIME_WEIGHT_MODS` их веса подписаны детально по WR (regime.py 50–110), а на
5м стоит немотивированный 1.10 — рассинхрон обоснованности между слоями.

## 7. M1/M2/M3: сейчас диагностика, потенциально — множитель веса кластера

Сейчас M1/M2/M3 (`cluster_models.py`) считаются (`compute`, 277–340) и пишутся в
запись сигнала (`sig["m1/m2/m3"]`, 7542–7544) **только для attribution/архива** —
в направление/размер не входят вообще. Из живого composite исключены сознательно
(6000–6007): они собраны из тех же base_scores, сложение = двойной счёт.

Но M1/M2/M3 — это **суб-композиты над РАЗНЫМИ группами кластеров**
(CLUSTER_GROUPS, 128–132): M1 = Цена/Тренд, M2 = Объём/Поток, M3 =
Позиционирование/Режим. **Их согласие/расхождение — реальная информация,
которую линейная сумма усредняет в ноль.** Сейчас похожую (но грубее) работу
делает `_divergence_score` (3509, только тренд-группа vs объём-группа) — M1/M2/M3
это тот же принцип, но по трём осям и взвешенный по effWR.

**Оценка ценности:** да, это недоиспользованный сигнал. Но включать их как
*аддитивный голос* или как множитель, поднимающий вес — вернёт ровно тот
двойной счёт, ради которого их вынесли. **Безопасная интеграция — как
модулятор уверенности/размера, а не как добавка к composite:**

- M1/M2/M3 согласны по знаку с composite → это подтверждение с трёх независимых
  осей → можно поднять confidence (осторожно, ближе к границе бакета).
- M1/M2/M3 расходятся между собой (напр. цена вверх, поток вниз) → дивергенция →
  **снизить** confidence/размер (мягкое вето), как расширение
  `_divergence_score`.

Это извлекает мета-информацию, не нарушая «без двойного счёта»: множитель на
confidence не добавляет новый голос в сумму, а модулирует итог. Конкретно —
завести `cluster_agreement = f(sign(M1),sign(M2),sign(M3), composite)` и подать
в `confidence_mult` рядом с rqa/wavelet (8348–8352), где уже живут такие
модуляторы. Начать с консервативного [0.8, 1.15] и калибровать по quality.

## Дополнительные находки (не входили в ТЗ)

1. **`_REGIME_METHOD_SIGN = {}` — мёртвая ветка.** Код 8253–8258 никогда не
   исполняется. Либо удалить (чистота), либо доделать per-ticker (комментарий 303
   говорит, что глобально не взлетело из-за AFKS≠AFLT) — тогда это правильный
   инструмент для п.2 (флип знака контр-тренда по режиму).

2. **`_EMPIRICAL_INVERTED_METHODS` — мёртвая константа.** Определена (250) с
   подробным обоснованием (KLINGER/DONCHIAN/PRICE_ACCEL, WR 35–40 % при согласии),
   но **нигде не применяется** — инверсия идёт только через UI
   (`set_inverted_methods`, dashboard 1641) или через выученный `IC.invert`.
   Т.е. задокументированная эмпирика не подключена автоматически; если IC не
   успел выучить инверсию — эти методы голосуют «как есть» вопреки статистике.
   Либо подключить frozenset к `_inverted_methods` при инициализации, либо снять
   вводящий в заблуждение комментарий.

3. **`_divergence_score` — ещё один слой ре-использования base-методов.**
   Подмешивается в linear_raw с весом 0.25 (8334–8337) из PRICE_TREND/TQ/ZLEMA vs
   VOL_MOMENTUM/KLINGER/VZO — все уже в сумме. Меньший масштаб, чем плейбуки, но
   концептуально то же (см. п.3/п.7: логичнее свести дивергенцию в один канал с
   M1/M2/M3, а не считать её в двух местах).

4. **Хрупкость step-функции размера.** Три жёстких порога (0.10/0.40/0.70)
   превращают непрерывный composite в 4 состояния. Любой из эффектов выше
   (двойной счёт, круглый MTF-вес) опасен ровно тем, что легко перебрасывает
   через порог. Стоит рассмотреть плавную (сигмоидную) `risk_pct(confidence)`
   вместо ступеней — тогда цена ошибок в весах перестанет быть «×2 размера».
